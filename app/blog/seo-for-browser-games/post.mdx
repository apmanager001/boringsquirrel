# Search traffic does not wait for polish

If a browser game only appears online once the full experience is complete, the launch has to do all the work at once. That is expensive and usually unnecessary.

What works better is a ladder:

1. Create the games hub.
2. Create individual game pages with clear promises.
3. Explain the mechanics and why the game is interesting.
4. Ship the interactive loop without changing the URL structure.

## What the page should already answer

For a search engine and a human reader, every early game page should answer the same questions:

- what is this game?
- what makes it different?
- can I play it now or is it coming soon?
- how will scoring or progression work?

<Callout title="Indexable before playable">
  A coming-soon page can still be useful if it contains real context, real copy,
  and a stable place in the site structure.
</Callout>

## Why this matters for this project

Sudoku, Oilcap, and AcornSweeper all have distinct intent behind them:

- **Sudoku** attracts classic puzzle search traffic.
- **Oilcap** has retro-game and strategy-puzzle angles.
- **AcornSweeper** can ride familiar Minesweeper language without looking generic.

That means the page architecture should already reflect those differences before the logic layers are finalized.
